# CodexTheme 哥特旗舰系列设计

## 决策摘要

codextheme.tech 首发内容从“九个主题槽位”收缩为三个可以实际安装、拥有真实 Codex 截图的哥特主题。三套主题属于同一原创世界观，复用可靠的 Codex UI 适配规则，但使用不同场景、材质和光线，不允许只做色相替换。

目标是在 2026-07-18 当天完成可安装主题、真机验证和网站展示。站方不使用未授权动漫、游戏角色或识别性标志。主视觉可以使用 AI 生成的原创环境，前提是经过人工艺术指导、筛选、界面适配和真实 Codex 验收。

用户已授权在不扩大范围的前提下自主完成本设计、执行计划与实现。普通视觉细节不再等待逐项确认；账号权限、系统授权或范围变化仍需要用户处理。

## 交付主题

### Cathedral Nocturne

- 中文说明：黑金圣堂。
- 核心场景：纵深巨大的幽暗圣堂内景、暖金窗光、黑曜石结构和少量深红织物。
- UI 材质：近黑半透明表面、旧金边框和低饱和暖白文字。
- 角色：三套中的首页旗舰，承担首屏主视觉。

### Crimson Procession

- 中文说明：赤月回廊。
- 核心场景：赤月下的修道院回廊、深红织物、黑色建筑剪影。
- UI 材质：近黑表面、暗红焦点色和克制的红色状态光。
- 角色：承担更强点击欲和暗黑动漫用户熟悉的情绪，但不复刻任何现有角色、服装、眼睛、图案或标志。

### Silver Reliquary

- 中文说明：苍银圣所。
- 核心场景：月光照亮的苍白遗迹、银灰石材、薄雾和冷色金属。
- UI 材质：冷灰半透明表面、苍银蓝焦点色和清晰的亮色文字。
- 角色：提供同系列中的冷静、低刺激选择。

三个英文名称是公开主名称。中文只作为简短解释，不为每套主题额外制造复杂世界观文案。

## 素材与权利边界

主题背景必须是原创环境，不包含人物、文字、Logo、商标、动漫角色、游戏角色或现成 IP 的识别性元素。不得下载网络壁纸或从第三方主题包提取素材。

AI 生成图只作为原始候选。候选进入主题前必须满足：

1. 没有可识别的现有作品、角色或品牌。
2. 没有明显文字、水印、错误标志或破碎建筑细节。
3. 构图服务于 Codex UI，而不是独立海报。
4. 有足够分辨率支持统一的桌面截图。
5. 在仓库素材来源清单中记录为站方生成的原创候选和最终采用版本。

站方 Gallery 只展示原创或有明确授权的主题。未来用户自己的动漫图片只允许走本地处理路径，不进入站方主题包、Gallery 或公共存储。社区投稿必须提供素材来源和再分发许可。

## Codex 构图与界面规则

每套主视觉都按 Codex Desktop 的 UI 安全区生成和裁切：

- 左侧约 25% 为低细节暗区，保证侧栏文字和项目列表可读。
- 中央保留首页标题、建议卡和 Session 正文的视觉空间。
- 建筑主体、窗口、月亮或高亮集中在右侧约 35%。
- 图片本身不绘制 Codex UI、按钮、文字或伪造窗口。
- Home 与 Session 可以使用同一场景的不同裁切和遮罩，以控制当天产量。

三套主题共享一个基础 CSS 层，用来定义背景承载、表面层级、文字、边框、焦点环和控件保护规则。每套主题只在独立变量和必要的局部选择器中覆盖背景、强调色、面板透明度和材质细节。

危险操作、禁用状态、焦点状态、发送按钮、输入行为和键盘语义保持 Codex 原生逻辑。主题不得为了视觉一致性隐藏系统反馈或降低正文可读性。

## 主题包与数据流

每套主题目录包含：

- `theme.json`：名称、slug、版本、作者、兼容性和资源声明。
- `assets/hero.jpg`：Home 背景。
- `assets/session.jpg`：Session 背景；允许从同一原图生成不同裁切。
- 共享 `themes/shared/codex.css` 与主题变量配置。

现有打包脚本生成 `.codextheme-theme` 包并复制到 CLI。Catalog 只把完成打包和真机验证的主题标记为 `available`。旧有 `midnight-circuit`、`crimson-eclipse` 和 `aurora-glass` 不再出现在首页和首发 Gallery；为避免破坏已经复制的命令，CLI 内部可以暂时保留其包和 slug。

用户路径保持：

```text
真实 Codex 截图 → 主题详情 → 复制固定版本 npx 命令 → 本地应用 → 可恢复官方外观
```

现有 `@codextheme/cli@0.1.0` 已发布且不能覆盖。三个新主题随 `@codextheme/cli@0.1.1` 发布，网站上的新安装命令固定到 `0.1.1`。旧主题已经复制出去的 `0.1.0` 命令继续有效。

安装失败必须保留应用前状态，不显示成功信息。结构不兼容时停止注入并提示恢复，而不是强行应用。

## 网站叙事与页面结构

首页不再展示主题槽位、制作数量或空卡片。结构调整为：

1. 首屏使用 `Cathedral Nocturne` 的真实 Codex Home 截图。
2. 标题直接表达结果，例如“把 Codex 变成你的工作世界”。
3. 首屏唯一主操作是复制或进入旗舰主题安装。
4. 下一屏展示三套真实主题卡片，图片占据卡片主要面积。
5. 每个详情页先展示 Home 和 Session 两张真实截图，再展示说明、兼容性、安装命令和恢复入口。
6. 安全说明保持简短，链接到完整安全页。

不展示虚假下载量、排行、评分或“正在制作”状态。没有真实截图的主题不发布。

`codextheme@codextheme.tech` 作为公开支持邮箱，出现在：

- 全站页脚。
- 安装帮助页。
- 安全说明页。

链接使用 `mailto:codextheme@codextheme.tech`。统一说明为“安装或兼容问题，请联系我们”。主题详情页不重复占据主操作位置，由页脚提供联系入口。

## 生产和验收流程

每套主题按相同门槛推进：

1. 生成多个原创环境候选。
2. 检查权利边界、构图、文字伪影和细节质量。
3. 选择并处理 Home 与 Session 资源。
4. 生成主题 manifest 和样式变量。
5. 打包并运行自动测试。
6. 在真实 Codex Desktop 应用。
7. 检查侧栏、首页卡片、正文、代码块、工具输出、输入框、焦点和危险状态。
8. 截取统一尺寸的 Home 与 Session 图片，清理私人数据。
9. 将验证通过的主题加入网站 Catalog。

如果某一套在当天无法通过可读性或真机兼容门槛，它不会以成品身份上线，也不会用合成截图替代。其余通过的主题可以独立发布，不要求三套同时放行。

## 测试与完成标准

自动验证至少覆盖：

- 三个新 slug 唯一且 catalog 字段完整。
- 每个 `available` 主题的 manifest、背景资源和打包产物存在。
- CLI 能识别并应用三个新 slug。
- 三个新主题的固定版本安装命令指向 `@codextheme/cli@0.1.1`。
- 网站首页只渲染拥有真实预览的首发主题。
- 主题详情、帮助、安全、robots 和 sitemap 正常构建。
- 联系邮箱在页脚、帮助页和安全页出现并使用 `mailto:`。
- 桌面和移动布局、键盘焦点与 reduced-motion 行为通过检查。

真机验收至少覆盖：

- Home 和 Session 的正文对比度足够。
- 侧栏、建议卡、项目选择器和输入框均可操作。
- 应用、重启后保持、切换主题和恢复官方外观成功。
- 截图不包含私人项目、路径、用户名或对话。

只有自动测试、构建和相关主题的真机验收完成后，才可以声称该主题“可用”并部署到生产站点。

## 明确不做

- 搜索、筛选、账号、收藏、评论、支付或数据库。
- 在线图片上传、在线编辑器或公共用户素材存储。
- 未授权动漫、游戏或影视 IP 主题。
- 人物型主题、动态背景、桌面宠物或复杂动画。
- 通过简单换色凑出主题数量。
- 使用合成 Codex 窗口替代真实截图。

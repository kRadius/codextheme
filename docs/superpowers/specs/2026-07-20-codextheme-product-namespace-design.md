# CodexTheme Product Namespace Design

## Goal

All newly generated, published, installed, and user-visible CodexTheme artifacts use the CodexTheme identity. CodeDrobe remains visible only where Apache-2.0 provenance or legacy migration requires it.

## Scope

This change covers the product boundary:

- published `@codextheme/cli` and `@codextheme/runtime` behavior;
- bundled catalog theme files;
- temporary private theme files generated by the website;
- renderer DOM classes, datasets, style ids, state keys, and image CSS variables created by a new apply;
- public error codes and CLI output;
- the managed Codex backup directory used by a new apply;
- website copy and install prompts.

It does not rewrite upstream history, remove Apache-2.0 attribution, or rename every symbol in legacy compatibility tests and migration code.

## Canonical Product Contract

New artifacts use the following names:

| Surface | Canonical value |
| --- | --- |
| Package format | `codextheme-theme` |
| Package extension | `.codextheme-theme` |
| Host root class | `codextheme-host-codex` |
| Skin profile class | `codextheme-codex-skin` |
| Renderer state | `globalThis.__CODEXTHEME__` |
| Theme style prefix | `codextheme-theme-style-` |
| Image variables | `--codextheme-image-<id>` |
| Theme dataset | `data-codextheme-theme` and `data-codextheme-theme-version` |
| Public errors | `CODEXTHEME_*` |
| New macOS backup directory | `~/Library/Application Support/CodexTheme/` |
| New Windows backup directory | `%LOCALAPPDATA%\CodexTheme\` |

The package extension already uses the CodexTheme identity. The package body, generated CSS, renderer ownership markers, and public errors must now match it.

## Compatibility Boundary

The runtime accepts existing `codedrobe-theme` packages as legacy input and normalizes them before applying. New pack, private generation, caching, and catalog build paths emit only `codextheme-theme`.

Before installing a new theme, the renderer cleanup removes both the new CodexTheme ownership markers and the historical CodeDrobe markers. Restore first uses the new CodexTheme backup path and falls back to the historical CodeDrobe backup path when no new backup exists. A successful migration consumes only the backup it restored.

Legacy names are allowed only in:

- `LICENSE`, `NOTICE`, `UPSTREAM.md`, and explicit provenance sections;
- a clearly named legacy compatibility module;
- migration fixtures and tests that prove old packages, renderer state, and backups remain recoverable.

They are not allowed in new theme packages, generated private packages, normal CLI stdout/stderr, website HTML, install prompts, current CSS sources, or current renderer state.

## Components

### Theme package normalization

The runtime defines one canonical CodexTheme format and one accepted legacy format. Readers accept both and return a canonical in-memory object. Writers and packers accept only source manifests and emit the canonical format. Validation continues to reject unknown formats and executable content.

### Renderer ownership

The renderer payload installs only CodexTheme-prefixed classes, datasets, state, styles, and image variables. Cleanup recognizes and removes both namespaces so a new apply replaces an existing legacy theme without requiring a manual restore.

The Codex compatibility profile uses CodexTheme-prefixed chrome ids and classes. It may query old ids only during cleanup and migration.

### Public errors and host state

Runtime errors are renamed to `CODEXTHEME_*`. The CLI lifecycle consumes only the renamed codes. Backup creation moves to the CodexTheme application-support directory. Restore retains a legacy fallback for existing users.

### Product generators

Catalog packages and private website packages emit the canonical package format and CodexTheme CSS namespace. Existing cached legacy private packages remain readable; reapply does not rewrite their bytes.

### Release guard

A repository check scans newly built product outputs and fails when `codedrobe` appears case-insensitively. The scan covers packed catalog themes, a generated private theme fixture, packaged CLI/runtime tarball contents excluding attribution documents, built website HTML, and captured normal CLI output.

## User Experience

Commands remain under `@codextheme/cli`. Existing catalog slugs and private links remain valid. Applying a legacy cached theme succeeds without exposing its old format name. Restore continues to recover users who applied a theme before this migration.

The only places a user should encounter the CodeDrobe name are the repository license, notice, and upstream provenance documents when they deliberately inspect the source and licensing history.

## Failure Handling

- A malformed legacy package fails with the same safe generic product message used for malformed current packages.
- If both new and legacy backup files exist, the new CodexTheme backup wins; the legacy file is not deleted.
- If renderer migration cleanup is incomplete, verification fails and apply rolls back rather than layering two controllers.
- Unknown package formats and unknown renderer profiles remain fail-closed.

## Testing

Tests are written before each behavior change and cover:

1. new pack output uses `codextheme-theme` and CodexTheme CSS variables;
2. old `codedrobe-theme` packages still resolve and apply;
3. new renderer payload contains no CodeDrobe namespace and removes historical markers;
4. public errors and CLI output contain no CodeDrobe name;
5. new backup paths are used while legacy backups remain restorable;
6. website private packages and bundled catalog packages contain no CodeDrobe name;
7. the release guard permits required attribution documents but rejects product output leakage;
8. the full CLI, runtime, theme, website, type, build, and package checks pass.

## Release

The migration is released as a coordinated runtime and CLI update because the canonical artifact format and renderer ownership contract change together. The CLI pins the new runtime version, rebuilds every bundled theme, and the website pins the new CLI command. Existing versions remain restorable through the new runtime's compatibility layer.
